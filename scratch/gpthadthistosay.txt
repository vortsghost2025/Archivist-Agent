# So yes — there *is* a way **we** should be doing things

And it looks like this:

## The implementation method

### Step 1: Separate principle from procedure

Do **not** put everything into AGENTS.md.

Use:

* `AGENTS.md` = constitution
* Rules = constraints
* Skills = reusable governance procedures
* Commands = user entry points
* Memory = curated precedent
* Subagents = scoped execution roles
* Permissions = hard blocks

That separation matches the current direction of Kilo, Codex, and Claude much better than a single-file governance blob. ([Kilo][2])

### Step 2: Design the agents around authority boundaries

You should not ask one general-purpose agent to “remember governance.”

Instead define roles like:

* **Constitutional reviewer**
  read-only, can inspect files, cannot edit

* **Archivist / memory keeper**
  can write only to governance memory folders

* **Executor**
  can edit code but cannot bypass preflight governance command

* **Planner**
  can reason and propose but cannot modify files

Kilo subagents support isolated sessions and per-agent permissions, which is exactly the right primitive for this. ([Kilo][4])

### Step 3: Default-deny dangerous behavior

In Kilo, use permissions as policy.

Examples:

* `bash = ask`
* `edit = ask`
* destructive shell actions = deny via workflow/rule policy
* `task = ask` so agent spawning is deliberate
* `skill = allow` only for approved governance skills

Kilo’s permission system supports allow/ask/deny at the tool level. ([Kilo][3])

### Step 4: Turn governance into commands, not reminders

Do not hope the model remembers to do the right process.

Make explicit commands:

* `/constitutional-check`
* `/drift-check`
* `/checkpoint`
* `/pre-merge-governance`
* `/incident-log`

Kilo workflows are built exactly for this pattern. ([Kilo][5])

### Step 5: Externalize event enforcement only where needed

If you need:

* automatic turn-end summaries
* automatic key scanning
* automatic validation on file edits
* automatic block/allow lifecycle actions

then you have two options:

#### Option A: external policy server through MCP

Use Kilo’s MCP support to call a policy/validator service. Kilo supports local and remote MCP servers in config. ([Kilo][6])

#### Option B: run the governance-critical workflows in Claude Code or Codex

Claude has explicit deterministic hooks; Codex has explicit hooks plus managed config layers. ([Claude][7])
